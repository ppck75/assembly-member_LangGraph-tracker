## 1. 프로젝트 개요

---

본 프로젝트는 사용자가 국회의원 이름을 입력하면 기본정보, 발의법안, 공동발의 네트워크, 본회의 표결 기록, 정당 다수 입장 일치도, 최근 언론 이슈를 한 화면에서 조회·분석할 수 있는 **LangGraph 기반 의정활동 추적 AI 에이전트**이다. 단순 의원 검색을 넘어 열린국회정보 API와 뉴스 데이터를 통합해 국회의원의 의정활동을 근거 중심으로 이해할 수 있도록 기획하였다. 웹 사이트에 접속해 의원명을 입력하면 발의법안 분석, 공동발의 네트워크, 표결 상세 조회, 정당 일치도 계산, 최근 뉴스 요약, 핵심 브리핑, 원자료 다운로드가 실행된다. 또한 발의법안 전용 Q&A 챗봇을 추가해, 사용자가 조회된 발의법안 데이터에 대해 후속 질문을 할 수 있도록 했다. 이때, 사용자는 사이드바에서 분석 범위와 LLM 사용 여부를 조정할 수 있다. 또한, 의원 목록 조회 기능도 함께 제공한다. 사용자는 대수별 의원 목록을 불러온 뒤 이름, 정당, 지역구, 위원회 정보를 확인하고 분석 대상을 선택할 수 있다. 이는 국회의원 명단이나 API 구조에 익숙하지 않은 일반 사용자도 검색 과정에서 막히지 않고 분석을 시작할 수 있도록 하기 위한 입력 보조 장치이다.

이 이미지는 LangGraph로 구현한 에이전트 워크플로우를 그래프 형태로 시각화한 것이다. 사용자의 의원명 입력이 분석 노드를 거쳐 최종 브리핑으로 병합되는 과정을 보여준다. 입력 정규화와 의원 기본정보 조회 후 발의법안 조회·표결 상세 조회·최근 이슈 검색을 병렬로 실행한다. 이후 입법 관심 분야, 공동발의 네트워크, 정당 다수 입장 일치도, 표결 해석 메모를 생성하고, 모든 결과를 `generate_activity_briefing` 노드에서 병합해 브리핑과 대시보드로 제공한다.

## 2. 프로젝트 필요성

---

### 2.1 의정활동 데이터의 접근성 문제와 개발 필요성

국회의원의 의정활동은 시민의 정치적 판단, 언론의 권력 감시, 시민단체의 정책 평가에 필요한 핵심 정보이다. 그러나 국회 관련 데이터가 '열린국회정보 API'로 제공되고 있더라도, 일반 시민이나 비전문 사용자가 API 구조를 이해하고 필요한 정보를 조합해 의정활동을 해석하기에는 여전히 높은 진입 장벽이 존재한다. 실제 데이터는 국회의원 기본정보, 발의법안, 의안별 표결현황, 의원별 본회의 표결정보처럼 여러 API 목록으로 흩어져 있기 때문에 특정 의원의 활동을 이해하려면 각 API에서 의원명을 따로 조회하고 표결값·정당·의안 정보를 다시 조합해야 한다. 특히 표결 불참, 정당 다수 입장 이탈, 분야별 발의 집중, 반복 공동발의 관계는 단일 자료만으로 파악하기 어렵다. 이러한 문제의식을 바탕으로 본 프로젝트는 공개된 의정활동 데이터를 단순 조회 가능한 자료로 남겨두지 않고, 시민·언론인·정책 연구자가 실제로 활용할 수 있는 분석 가능한 정보로 전환하여 '데이터 공개'와 '데이터 활용'사이의 간극을 줄이는 것을 목표로 한다. 

본 프로젝트의 분석 대상인 **'국회의원 의정활동(legislative activity)'**은 국가와 지역에 관련된 다양하고 복합적인 사회적 현안·의제를 포함 할 뿐 아니라 이와 관련된 의원들의 관점과 개성이 연결된 결과물이다. 이재윤·김의진·박태선(2024)에 따르면 국회의원은 국회라는 합의체의 구성원, 국민 전체의 대표자, 정당정치에 참여하는 행위자라는 복합적 지위를 갖기 때문에, 발의법안과 표결 기록은 단순 행정 데이터가 아니라 의원이 어떤 의제에 관심을 두고 어떤 판단을 반복해왔는지 보여주는 정치적 기록으로 연구적 가치를 지닌다. 동시에 국회의원 의정활동은 시민이 대표자의 활동을 평가하고 정치적 판단을 형성하기 위해 지속적으로 확인해야 하는 공적 정보이기도 하다. 국회의원이 어떤 법안을 발의하고, 어떤 표결에 참여 하거나 불참했으며, 정당의 다수 입장과 어떤 관계를 보였는지는 선거 이후에도 시민이 대표자를 감시하고 책임성을 요구하는 데 필요한 핵심 근거가 된다. 본 프로젝트는 공개되어 있지만 흩어져 있는 제도정치 데이터를 시민이 바로 이해할 수 있는 정보로 재구성해 정치 정보 접근성과 데이터 기반 감시저널리즘을 지원한다. 이러한 점에서 본 프로젝트는 단순 조회 서비스를 넘어 제도정치와 시민사회 사이의 정보 격차를 줄이는 정치 커뮤니케이션 도구로 기능할 수 있다. 

이러한 문제의식을 사용자가 함께 이해할 수 있도록, 웹 페이지 사이드바에 **이론 배경** 영역을 구성했다. 이 영역은 의정활동의 개념, 의정활동 기록의 의미, 공론장과 정치 커뮤니케이션, 감시저널리즘과 민주적 책임성, 국회의원의 복합적 지위, 정당 다수 입장 일치도의 중요성을 접고 펼칠 수 있는 설명으로 제공한다. 따라서 사용자는 단순히 수치와 표를 확인하는 데 그치지 않고, 왜 발의법안·표결·공동발의·정당 일치도 같은 지표가 국회의원 의정활동을 이해하는 데 중요한지 이론적 맥락과 함께 파악할 수 있다.


### 2.2 참신성

기존 국회 정보 서비스가 API별·자료 유형별 조회에 가까웠다면, 본 프로젝트는 **개별 의원 중심의 통합 의정활동 분석**을 제공한다. 한 의원을 기준으로 기본정보, 발의법안, 본회의 표결, 정당 다수 입장 일치도, 공동발의 네트워크, 최근 이슈를 하나의 워크플로우 안에서 통합 분석한다. 

본 프로젝트의 참신성은 다섯 가지이다. 첫째, 발의법안과 본회의 표결 기록을 하나의 의원 프로필 안에서 연결해, 특정 의원이 어떤 법안을 발의했고 어떤 표결 선택을 반복했는지 함께 확인할 수 있게 한다. 둘째, 공동발의 관계를 ego-network로 시각화해 반복 협업 의원, 당내·당외 협업 비율, 협업 소관위원회를 한 눈에 제시한다. 셋째, 공식 당론을 직접 판정하지 않고 같은 정당 의원들이 실제로 가장 많이 선택한 표결값을 정당 다수 입장으로 계산해, 대상 의원의 일치·이탈·불참을 분류한다. 넷째, 열린국회정보 API 기반 공식 의정활동 데이터와 Google News RSS 기반 최근 이슈를 구분해 함께 제공함으로써 제도적 활동과 공적 보도 맥락을 동시에 파악할 수 있게 한다. 다섯째, LangGraph 기반 **Parallel Workflow**를 적용해 서로 의존하지 않는 발의법안 조회, 표결 상세 조회, 최근 뉴스 검색을 병렬 실행한다. 여기에 발의법안 데이터 기반 **Q&A 챗봇**을 결합해 사용자가 분석 결과에 대해 추가 질문을 할 수 있도록 한다. 이는 단순 챗봇을 넘어 공공데이터 수집·계산·해석·시각화·질의응답을 결합한 AI 에이전트라는 점에서 차별화된다.

### 2.3 활용 방안

본 서비스는 시민, 언론인, 시민단체, 연구자, 정치·저널리즘 교육 현장에서 활용될 수 있다.

- **시민**은 의원의 이미지나 정쟁적 발언이 아니라 발의, 표결, 정당 일치도, 최근 이슈라는 객관적 자료를 바탕으로 의정활동을 살펴볼 수 있다.
- **언론인**은 표결 이탈, 불참 비중, 반복 공동발의 관계, 특정 정책 분야에 집중된 입법 활동을 확인해 후속 취재의 단서를 얻을 수 있다.
- **시민단체와 정책 감시 조직**은 특정 의제나 위원회 중심으로 의원별 활동을 비교하는 기초 자료로 활용할 수 있다.
- **정치 커뮤니케이션·저널리즘 교육**에서는 공공데이터 기반 감시저널리즘, 데이터 저널리즘, 공론장 정보 매개 장치의 사례로 활용할 수 있다.

## 4. 분석·개발 방법론

---

### 4.1 시스템 구조와 기술스택

시스템은 **Streamlit 웹 인터페이스**, **LangGraph 워크플로우**, **열린국회정보 API 수집 모듈**, **LLM 해석 모듈**, **캐시·오류 격리 계층**, **시각화 컴포넌트**로 구성된다. 핵심 파일은 `app.py`(UI와 탭 렌더링), `src/member_activity_workflow.py`(LangGraph 노드, API 호출, 캐시, LLM 분석, 표결·정당 일치도 계산), `src/config.py`(API 키와 환경 변수), `src/member_directory.py`(의원 목록·정당 매칭), `src/streamlit_runtime.py`(클라이언트·그래프 재사용)이다.

주요 기술스택은 Python(분석 로직), Streamlit(웹 UI·배포), LangGraph(워크플로우 제어·병렬 실행), pandas(데이터 정리·통계), requests(동기 API 호출), httpx(비동기 API 호출)이다. 시각화에는 Streamlit 기본 UI와 JavaScript의 ECharts, vis-network CDN을 사용하고, LLM 모델은 `gemini-2.5-flash-lite`로 고정한다. 열린국회정보 API 키는 서버 측 환경 변수로 관리하고, 사용자는 Gemini API 키를 직접 입력해 LLM 분석을 사용할 수 있다.

웹 UI는 사용자가 분석 목적과 대기 가능 시간에 따라 조회 범위를 조정할 수 있도록 설계했다. 사이드바에서는 발의법안, 표결 상세 조회, 정당 일치도, 최근 이슈, LLM 해석 사용 여부를 선택할 수 있으며, 표결 상세 조회와 정당 일치도 분석은 최근 N건 기준으로 범위를 조정할 수 있다. 이를 통해 사용자는 빠른 탐색이 필요한 경우에는 일부 기능을 끄거나 범위를 줄이고, 더 깊은 분석이 필요한 경우에는 조회 범위를 확장하는 방식으로 속도와 분석 깊이를 조절할 수 있다.

### 4.2 LangGraph 워크플로우와 성능 개선

본 프로젝트는 LLM이 임의로 도구 호출 순서를 정하는 챗봇이 아니라, `StateGraph(MemberActivityState)`로 노드와 의존 관계를 코드 수준에서 정의한 에이전트이다. 전체 흐름은 선행 실행 이후 서로 독립적인 작업을 병렬로 분기하고, 각 분기 분석이 끝난 뒤 결과를 다시 합치는 Parallel Workflow 구조이다. 그래프 내 노드의 실행 방식과 역할을 다음과 같다. 

| 실행 구간 | 노드명 | 실행 방식 | 역할 |
|:---|:---|:---|:---|
| 1 | `normalize_user_request` | 선행 실행 | 사용자 요청 및 분석 옵션 정규화 |
| 2 | `get_member_info` | 선행 실행 | 의원 기본정보, 재임 대수, 분석 가능 대수 산출 |
| 3-A | `search_member_bills` | 병렬 실행 | 대표발의·공동발의 법안 수집 및 처리결과/소관위원회 통계 계산 |
| 3-B | `get_all_member_votes` | 병렬 실행 | 본회의 표결 상세 조회 및 찬성·반대·기권·불참 분류 |
| 3-C | `search_recent_member_news` | 병렬 실행 | Google News RSS 기반 최근 이슈 수집 및 요약 |
| 4-A-1 | `analyze_legislative_interests` | 병렬 실행 | 발의법안 기반 입법 관심 분야 분석 |
| 4-A-2 | `analyze_cosponsor_network` | 병렬 실행 | 공동발의 파트너 및 협업 패턴 분석 |
| 4-B-1 | `analyze_party_alignment` | 순차 실행 | 같은 정당 의원 다수 입장과의 일치·이탈 분석 |
| 4-B-2 | `interpret_vote_statistics` | 순차 실행 | 조회 범위와 불참 비중을 고려한 표결 해석 메모 생성 |
| 5 | `generate_activity_briefing` | Fan-in 이후 실행 | 입법·공동발의·표결·최근 이슈를 종합한 핵심 브리핑 생성 |
| 6 | `summarize_member_activity` | 최종 실행 | 화면 표시용 요약 결과 정리 |

초기 구현은 모든 노드를 Sequential Workflow로 연결해 순차 처리했다. 구조는 단순했지만 API 호출, LLM 해석, 통계 계산이 차례로 대기하면서 기본 설정값(최근 재임 대수, 표결 상세 100건, 정당 일치도 100건) 기준 약 1분 30초가 소요되었다. 이처럼 기본 설정값임에도 불구하고 분석 결과 산출에 상닫한 시간이 필요했으며, 설정 값을 조절하면 분석 시간이 배로 늘어나 병목 현상을 해결할 필요가 있었다. 성능 병목이 가장 크게 발생한 지점은 표결 상세 조회와 정당 다수 입장 일치도 분석이었다. 두 기능은 먼저 의안별 표결현황 API에서 `BILL_ID` 목록을 확보한 뒤, 각 `BILL_ID`에 대해 국회의원 본회의 표결정보 API를 반복 호출해야 한다. 예를 들어 표결 상세 범위를 100건으로 설정하면 최대 100개의 `BILL_ID`별 상세 조회가 필요하고, 정당 일치도 분석 역시 각 표결에서 같은 정당 의원들의 표결 분포를 다시 확인해야 한다. 이 반복 호출을 순차적으로 처리하면 API 응답 대기 시간이 그대로 누적된다. 

베타 버전에서는 이러한 문제를 **Parallel Workflow**와 **비동기 I/O 처리**로 해결하였다. 그 결과 기존 순차 실행 방식에서는 1분 30초 내외가 소요되던 초기 분석 시간을 약 1분 줄인 30초 내외로 단축하여, 사용자가 웹 환경에서 기다릴 수 있는 수준의 응답성을 확보하였다. 구체적인 적용 방식은 다음과 같다. 

- LangGraph의 상위 흐름은 **fan-out/fan-in 구조**로 설계했다. 상위 그래프에서는 `get_member_info` 이후 `search_member_bills`, `get_all_member_votes`, `search_recent_member_news`가 fan-out되어 동시에 실행되고, 각 분기 결과는 `generate_activity_briefing`에서 fan-in된다. 병렬화 기준은 서로의 산출물을 기다릴 필요가 없고, API 호출이나 중간 계산 작업이 서로 겹치지 않는 노드만 동시에 실행하는 것이다. 발의법안 조회와 표결 상세 조회는 의원 기본정보만 필요하지만 서로 다른 데이터 조회 흐름을 사용하므로 동시에 실행하고, 정당 다수 입장 일치도는 표결 상세 조회 이후에만 계산한다. 

- 하위 구현에서는 `src/member_activity_workflow.py`에 `AsyncAssemblyAPIClient`를 추가해 `httpx.AsyncClient` 기반으로 열린국회정보 API를 호출한다. 구현 방식은 `async def`로 비동기 함수를 정의하고, 그 내부에서 `await`로 API 응답 대기를 처리하는 구조이다. `asyncio.Semaphore`로 동시 요청 수를 제한하고, `asyncio.as_completed()`로 완료된 요청부터 결과를 수집한다. 이 방식은 표결 상세 조회와 정당 일치도 분석에서 여러 `BILL_ID` 요청을 동시에 진행해 네트워크 대기 시간을 줄인다. 기존에는 `BILL_ID`를 하나씩 또는 스레드 병렬로 처리했다면, 현재는 비동기 클라이언트가 여러 요청을 동시에 보내고, 실패한 요청은 개별 실패로 격리한다. 만약 비동기 호출 자체가 실패하면 기존 `ThreadPoolExecutor` 기반 병렬 조회로 fallback되므로, 성능 개선을 시도하면서도 웹서비스의 안정성을 유지한다. 특히, 초기 표결 상세 조회 범위나 정당 일치도 분석 범위를 기본값보다 크게 늘리더라도 여러 `BILL_ID` 요청을 동시에 처리하므로 분석 시간이 요청 수에 비례해 급격히 증가하는 문제를 완화했다. 이를 통해 사용자가 더 넓은 범위의 표결 데이터를 조회하더라도 기존 순차 처리 방식보다 안정적인 응답 시간을 확보할 수 있다.

추가로 의원 목록은 Streamlit 캐시로, 발의법안·표결 의안 목록·의원별 표결·BILL_ID별 정당 표결 분포·LLM 결과·최근 뉴스는 파일 캐시로 저장한다. 기본 TTL은 24시간이며 최근 뉴스는 6시간 기준으로 관리한다. 특히 표결 상세 조회와 정당 일치도 분석은 `BILL_ID` 단위 캐시를 공유하므로, 같은 의원을 다시 분석하거나 정당 일치도 범위만 바꿔 재분석할 때 이미 조회한 표결 상세를 재사용할 수 있다. 표결 수백 건 중 일부 호출이 실패해도 부분 결과를 제공하고, 실패분은 다음 실행에서 재시도할 수 있게 설계했다. 비동기 처리와 캐시를 결합함으로써 첫 실행에서는 네트워크 대기 시간을 줄이고, 반복 실행에서는 API 호출 자체를 줄이는 방식으로 성능 문제를 완화한다.

분석 과정이 수십 초 이상 걸릴 수 있다는 점을 고려해, 웹 화면에는 현재 실행 중인 단계와 완료된 단계를 진행률 형태로 표시한다. 사용자는 의원 기본정보 조회, 발의법안 조회, 표결 상세 조회, 정당 일치도 분석, 최근 이슈 수집, LLM 해석 중 어느 단계가 진행 중인지 확인할 수 있다. 이는 공공 API 호출과 LLM 분석이 결합된 워크플로우에서 사용자가 시스템이 멈춘 것으로 오해하지 않도록 하기 위한 사용성 장치이다.

### 4.3 주요 분석 기능

- **의원 기본정보 및 재임 대수 필터링**은 전체 분석의 기준점이다. 사용자가 의원명을 직접 입력하면 먼저 국회의원 정보 통합 API에서 정당, 선거구, 위원회, 재임 대수를 확인하고, 이후 분석 범위를 해당 의원이 실제 활동한 대수로 제한한다. 동시에 특정 의원을 미리 정해두지 않은 사용자도 대수·정당·지역·위원회 조건으로 의원 목록을 탐색할 수 있도록 `ALLNAMEMBER` 기반 의원 카드를 제공한다. 사용자는 카드에서 의원의 상세 정보를 비교하며 분석 대상을 선택할 수 있고, 선택한 의원명은 입력창에 자동 반영된다. 이를 통해 사용자가 직접 검색하는 방식과 목록을 탐색하며 분석 대상을 결정하는 방식을 모두 지원힌다. 

- **요약** 탭에서는 의원 프로필, 발의법안 통계, 공동발의 네트워크, 표결 요약, 정당 일치도 요약, 최근 뉴스 수를 결합한 **핵심 브리핑**을 첫 화면에 제공한다. 

- **발의법안 분석과 입법 관심 분야 해석**은 의원별 발의법안 API를 기반으로 대표발의와 공동발의를 구분하고, 처리결과와 소관위원회별 통계를 계산한다. LLM은 발의법안 목록과 위원회 분포를 바탕으로 입법 관심 분야를 요약하되, 정치적 호불호나 성과 평가가 아니라 법안명과 통계에 근거한 제한적 해석만 생성하도록 설계했다.

- **공동발의 네트워크 분석**은 대상 의원과 같은 법안에 이름을 올린 의원들을 연결하는 ego-network로 구성한다. 공동발의 빈도에 따라 노드 크기를 조정하고, 협업 관계를 당내·당외·정당 미확인으로 구분하며, 반복 협업 의원과 협업 소관위원회도 함께 제시한다. 다만 공동발의가 정책 동의나 친소관계를 직접 증명하는 것은 아니므로, 법안 발의 과정에서 나타난 입법 협력 관계를 탐색하는 지표로 해석한다. 추가로 '반복 협업 의원 Top 10'을 바 차트로 시각화하였고, '파트너 상세 보기' 필터링을 통해 특정 의원과 함께 발의안 법안의 상세정보를 직접 조회할 수 있도록 하였다. 

- **발의법안 Q&A 챗봇**은 전체 워크플로우 결과 중 발의법안 데이터만 별도 context로 사용한다. 사용자는 “최근 발의한 법안 5개”, “가장 많이 발의한 분야”, “대표발의와 공동발의 비중”, “통과된 법안 여부”처럼 발의법안 탭의 결과를 바탕으로 추가 질문을 할 수 있다. 베타 버전의 챗봇은 표결, 뉴스, 정당 일치도 질문에는 답하지 않고 해당 탭을 확인하도록 안내하며, 답변에는 `판단 기준`과 `확인 자료`를 구분해 제시한다. 최근 법안 목록처럼 단순 조회가 더 적합한 질문은 LLM을 거치지 않고 규칙 기반으로 처리해 불필요한 해석과 반복 문구를 줄였다.

- **표결 상세 조회**는 `ncocpgfiaoituanbr`와 `nojepdqqaweusdfbi`를 결합한다. 의안별 표결현황 API에서 표결 대상 의안과 `BILL_ID`를 확보하고, 의원별 본회의 표결정보 API에서 찬성·반대·기권·불참 기록을 수집한다. 기본 범위는 최근 100건이며 사용자가 조정할 수 있다. 먼저 의원명 기준 일괄 조회를 시도하고, 부족한 경우 `BILL_ID`별 조회로 보완한다. `BILL_ID`별 조회 구간은 비동기 병렬 처리로 실행해 여러 표결 상세 요청을 동시에 보내며, 캐시된 `BILL_ID`는 API를 다시 호출하지 않는다. 표결 조회는 일부 API 실패가 있어도 성공·실패 건수를 분리해 전체 워크플로우가 중단되지 않도록 한다. 또한 표결이 진행된 날짜, 소관위원회, 국회 의안정보 시스템의 '의안상세' url 주소 등 표결에 대한 상세 정보를 함께 조회할 수 있도록 하였다. 

- **정당 다수 입장 일치도 분석**은 공식 당론 판정이 아니다. 각 표결에서 같은 정당 의원들이 실제로 가장 많이 선택한 찬성·반대·기권 값을 정당 다수 입장으로 계산하고, 대상 의원의 표결이 같으면 일치, 다르면 이탈, 대상 의원이 표결하지 않았으면 불참으로 분류한다. 무소속, 정당 내 유효 표결 없음, 찬반기권 동률은 판정 제외로 처리한다. 이 분석도 `BILL_ID`별 정당 표결 분포를 반복 조회해야 하므로 비동기 병렬 처리를 적용했다. 이미 표결 상세 조회에서 확보한 의원별 표결 row와 `BILL_ID`별 캐시를 재사용하고, 정당 일치도 범위만 재분석할 때는 전체 워크플로우를 다시 실행하지 않고 정당 일치도 결과만 갱신한다. 이 지표는 의원 개인의 표결 행동을 정당정치의 맥락 속에서 이해하기 위한 보조 지표이다.

- **최근 이슈 요약과 핵심 브리핑**은 Google News RSS에서 검색한 기사 메타데이터를 기반으로 한다. LLM은 제공된 검색 결과만 근거로 최근 이슈를 요약하며, 기사 내용의 사실관계나 의원의 의도를 단정하지 않는다. 

### 4.4 AI 해석의 검증 가능성과 왜곡 방지 설계

 AI hallucination과 왜곡 가능성을 줄이기 위한 검증 장치도 함께 설계했다. 특히, 국회의원 의정활동 데이터는 특정 의원과 정당에 대한 정치적 평가로 이어질 수 있는 정보이므로, 본 프로젝트는 LLM 해석의 신중성과 검증 가능성을 핵심 설계 원칙으로 두고 AI 편향 문제를 보완하기 위해 다음과 같은 구조를 적용했다. 첫째, LLM 입력은 원자료 전체가 아니라 발의법안·표결·공동발의·뉴스 검색 결과를 구조화한 JSON context로 제한해 모델이 임의의 배경지식을 끌어오지 않도록 했다. 둘째, 시스템 프롬프트에서 정치적 의도, 성과, 호불호를 단정하지 못하도록 제한하고 “가능성”, “신호”, “해석상 주의”처럼 조심스러운 표현을 사용하게 했다. 셋째, LLM 해석에는 `판단 기준`과 `확인 자료`를 제시하도록 했다. 이를 통해 사용자는 LLM의 해석이 어떤 기준에서 나온 것인지, 어떤 법안·표·뉴스 URL을 확인하면 되는지 분리해서 볼 수 있다. 넷째, 열린국회정보 API 기반 공식 의정활동 데이터와 Google News RSS 기반 웹 검색 맥락을 화면에서 구분해, 사용자가 어떤 정보가 공식 데이터이고 어떤 정보가 외부 보도 기반인지 확인할 수 있게 했다. 다섯째, 최근 이슈 요약에는 분석 근거 뉴스 목록과 URL을 함께 제공해 LLM 요약의 근거를 사용자가 직접 대조할 수 있게 했다. 여섯째, 원자료 탭에서 전체 결과 JSON, CSV 다운로드, 오류·확인 메시지를 제공해 LLM 해석과 원자료의 일치 여부를 검증할 수 있게 했다. 일곱째, 정당 다수 입장 일치도는 공식 당론이 아니라 같은 정당 의원들의 실제 다수 표결값이라는 점을 분명히 명시해, 사용자가 지표를 과도하게 해석하지 않도록 했다. 

## 5. 베타 구현 현황

---

베타 버전은 Streamlit 웹으로 배포한 상태이며, 요약 탭, 발의 법안 탭, 발의법안 Q&A 챗봇 탭, 표결 탭, 정당 일치도 탭, 최근 이슈 탭, 원자료 탭을 제공한다. 각 탭에서는 의원 프로필과 핵심 브리핑, 발의 통계와 공동발의 네트워크, 발의법안 기반 후속 질의응답, 찬성·반대·기권·불참 분류, 정당 일치도 요약과 이탈 표결 목록, Google News RSS 기사 목록, 워크플로우 JSON과 오류 메시지를 확인할 수 있다. 또한 사이드바에는 의원 목록 조회, 분석 옵션 설정, 이론 배경 설명을 배치해 분석 대상 선택, 분석 범위 조정, 결과 해석의 맥락 이해가 한 화면 안에서 이어지도록 구성했다.

## 6. 발전 방향 및 기대 효과

---

향후 핵심 개선 방향은 표결 상세 조회를 최근 N건 기준에서 **본회의 날짜 기준**으로 확장하는 것이다. 최근 N건 방식은 하루에 많은 표결이 몰릴 경우 특정 날짜의 맥락이 일부만 반영되고, 의원이 전부 불참한 본회의 날짜가 잘 드러나지 않는 한계가 있다. 날짜 기반 방식은 의안별 표결현황 API에서 본회의 날짜 목록을 만들고, 사용자가 선택한 날짜의 전체 `BILL_ID`를 기준으로 찬성·반대·기권·불참과 정당 다수 입장 일치도를 다시 계산한다. 기본값은 최근 N건으로 유지하되, 사용자가 “날짜 후보 불러오기” 버튼을 눌렀을 때만 후보를 조회한다.

추가 고도화 방향으로는 의원 간 비교 기능, 정책 키워드 기반 발의·표결 추적, 소관위원회와 의정활동 연계, 뉴스 관련성 필터 고도화, Markdown·PDF·CSV 리포트 내보내기, API 제공 범위 안내 강화가 있다. 챗봇 기능은 현재 발의법안 탭 전용 베타 기능이므로, 향후에는 표결·정당 일치도·공동발의 네트워크까지 질의응답 범위를 확장할 수 있다. 다만 정치적 해석 오류를 줄이기 위해 모든 챗봇 답변은 탭별 데이터 범위 안에서만 작동하게 하고, 답변마다 판단 기준과 확인 자료를 분리해 보여주는 방식을 유지할 계획이다.

본 프로젝트는 국회의원 의정활동을 데이터 기반으로 감시하고 이해할 수 있는 기반을 제공한다. 시민은 발의법안과 표결을 빠르게 확인하고, 정당 다수 입장과의 관계를 통해 표결 행동의 맥락을 살펴볼 수 있다. 언론인은 반복 공동발의, 특정 분야 발의 집중, 높은 불참 비중, 정당 다수 입장 이탈 표결을 취재 단서로 활용할 수 있다. 기술적으로는 LangGraph를 활용해 공공데이터 수집, fan-out/fan-in Parallel Workflow, 비동기 API 호출, 캐시, 오류 격리, LLM 기반 해석을 결합한 실용적 AI 에이전트 사례를 제시한다.
