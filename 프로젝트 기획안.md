## 1. 프로젝트 제목

**국회의원 의정활동 추적 AI 에이전트: 발의법안·표결·정당 다수 입장 일치도·공동발의 네트워크를 통합 분석하는 데이터 기반 정치 커뮤니케이션 웹서비스**

본 프로젝트는 사용자가 특정 국회의원 이름을 입력하면 해당 의원의 기본정보, 발의법안, 공동발의 네트워크, 본회의 표결 기록, 정당 다수 입장 일치도, 최근 언론 이슈를 한 번에 조회하고 분석할 수 있는 **LangGraph 기반 국회의원 의정활동 추적 AI 에이전트**를 개발하는 것을 목표로 한다. 단순한 의원 정보 검색을 넘어, 산재된 열린국회정보 API와 최근 뉴스 데이터를 시민과 언론인이 이해하기 쉬운 형태로 재구성하고, 국회의원의 의정활동을 근거 중심으로 살펴볼 수 있도록 지원한다.

현재 본 프로젝트는 베타 버전이 구현된 상태이다. Streamlit 웹 애플리케이션에서 의원명을 입력하면 발의법안 분석, 공동발의 네트워크, 표결 상세 조회, 정당 다수 입장 일치도 분석, 최근 뉴스 요약, 핵심 브리핑, 원자료 다운로드가 실행된다. 사용자는 세부 설정을 통해 발의법안 조회 범위, 표결 조회 범위, 정당 일치도 분석 범위, LLM 분석 사용 여부를 조정할 수 있다.

## 2. 프로젝트 필요성

### 2.1 사회적 문제: 공개된 의정활동 데이터의 산재와 해석 장벽

국회의원의 의정활동은 민주주의에서 시민의 정치적 판단과 언론의 감시 기능을 가능하게 하는 핵심 정보이다. 그러나 실제 의정활동 데이터는 여러 API와 웹페이지에 흩어져 있으며, 시민이 이를 직접 수집하고 해석하기에는 진입장벽이 높다. 특정 의원의 발의법안을 보려면 발의법안 API를 조회해야 하고, 본회의 표결 기록을 보려면 별도의 표결정보 API를 사용해야 한다. 또한 특정 의원의 표결이 개인적 판단인지, 같은 정당 의원들의 다수 입장과 일치한 선택인지 파악하려면 의안별 전체 의원 표결 데이터를 다시 조합해 계산해야 한다.

즉, 데이터는 공개되어 있지만 시민이 바로 이해 가능한 정보로 제공되지는 않는다. 이로 인해 국회의원의 의정활동은 종종 단편적 기사, 정쟁적 발언, 이미지 중심 평가에 의해 소비되고, 실제 법안 발의와 표결 기록에 기반한 평가는 어렵다. 특히 표결 불참, 정당 다수 입장 이탈, 특정 분야에 집중된 발의 활동, 반복적인 공동발의 관계 같은 정보는 의정활동을 이해하는 데 중요하지만 여러 데이터셋을 결합하지 않으면 파악하기 어렵다.

본 프로젝트는 이러한 문제를 해결하기 위해 **제도정치의 데이터를 시민사회가 이해 가능한 정보로 재구성하는 도구**를 개발한다. 사용자는 의원명만 입력하면 발의법안, 표결, 정당 일치도, 공동발의 관계, 최근 이슈를 하나의 화면에서 확인할 수 있다. 이는 정치 정보 접근성을 높이고, 시민과 언론인이 국회의원을 평가할 수 있는 근거를 제공한다는 점에서 필요성이 있다.

### 2.2 참신성: 개별 의원 중심의 통합 의정활동 분석

기존의 국회 정보 서비스는 대체로 API 목록별·자료 유형별 조회에 가깝다. 반면 본 프로젝트는 **개별 의원 중심의 활동 추적**을 목표로 한다. 의원을 기준으로 여러 데이터 소스를 묶고, 단순 목록 제공을 넘어 통계, 네트워크, 표결 패턴, 정당 다수 입장과의 관계를 함께 제시한다.

본 프로젝트의 참신성은 다음과 같다.

1. **발의법안과 표결 기록을 하나의 의원 프로필 안에서 연결**한다.  
   사용자는 특정 의원이 어떤 법안을 발의했는지와 본회의 표결에서 어떤 선택을 반복했는지 함께 확인할 수 있다.

2. **공동발의 네트워크를 시각화**한다.  
   법률안을 발의하려면 의원 10인 이상의 찬성이 필요하며, 발의 명단은 보통 대표발의자와 공동발의자로 구성된다. 따라서 공동발의 관계는 법안 발의 요건을 충족하는 과정에서 형성된 입법 협력 관계로 볼 수 있다. 본 프로젝트는 공동발의 관계를 1차 ego-network로 시각화하여 반복 협업 의원, 당내·당외 협업 비율, 협업 소관위원회를 보여준다.

3. **정당 다수 입장 일치도 분석을 제공**한다.  
   본 프로젝트는 공식 당론을 직접 판정하지 않는다. 대신 각 표결에서 같은 정당 의원들이 실제로 가장 많이 선택한 표결값을 정당 다수 입장으로 계산하고, 대상 의원의 표결이 그 다수 입장과 일치했는지, 이탈했는지, 불참했는지를 분류한다. 이를 통해 의원 개인의 표결 행동을 정당정치의 맥락 속에서 살펴볼 수 있다.

4. **최근 뉴스 이슈와 공식 의정활동 데이터를 구분해 함께 제공**한다.  
   Google News RSS 기반 최근 이슈는 열린국회정보 API 기반 공식 의정활동 데이터와 분리하여 제공한다. 이를 통해 사용자는 의원의 제도적 활동과 최근 공적 보도 맥락을 함께 볼 수 있다.

5. **LangGraph 기반 병렬 워크플로우를 적용**한다.  
   의원 기본정보를 확인한 뒤 서로 의존하지 않는 발의법안 조회, 표결 상세 조회, 최근 뉴스 검색을 병렬로 실행한다. 이는 단순 챗봇이 아니라 공공데이터 수집·계산·해석·시각화를 결합한 데이터 에이전트 구조라는 점에서 차별화된다.

### 2.3 활용 방안

본 프로젝트는 시민, 언론인, 시민단체, 연구자, 정치 교육 현장에서 활용될 수 있다.

- **시민**은 의원의 이미지나 정쟁적 발언이 아니라 발의, 표결, 정당 일치도, 최근 이슈라는 객관적 자료를 바탕으로 의정활동을 평가할 수 있다.
- **언론인**은 표결 이탈, 불참 비중, 반복적 공동발의 관계, 특정 정책 분야에 집중된 입법 활동을 확인해 후속 취재의 단서를 얻을 수 있다.
- **시민단체와 정책 감시 조직**은 특정 의제나 위원회 중심으로 의원별 활동을 비교하는 기초 자료로 활용할 수 있다.
- **정치 커뮤니케이션·저널리즘 교육**에서는 공공데이터 기반 감시저널리즘, 데이터 저널리즘, 공론장 정보 매개 장치의 사례로 활용할 수 있다.

하버마스(2006)는 공론장을 국가와 사회 사이를 매개하는 의사소통 체계로 설명한다. 숙의 민주주의가 작동하려면 시민이 공적 쟁점에 접근하고 이해할 수 있어야 한다. Strömbäck(2005)은 민주주의와 저널리즘의 관계를 사회계약으로 설명하며, 저널리즘은 시민의 자율적 판단을 위한 정보 제공과 권력 감시 기능을 수행한다고 본다. 이러한 관점에서 본 프로젝트는 단순한 정보 조회 서비스가 아니라, 제도정치와 시민사회 사이의 정보 격차를 줄이는 데이터 기반 정치 커뮤니케이션 도구로 기능할 수 있다.

## 4. 분석·개발 방법론

### 4.1 시스템 구조와 기술스택

본 프로젝트는 **Streamlit 기반 웹 인터페이스**, **LangGraph 기반 워크플로우**, **열린국회정보 API 수집 모듈**, **LLM 기반 해석 모듈**, **캐시 및 오류 격리 계층**, **시각화 컴포넌트**로 구성된다.

핵심 구현 파일은 다음과 같다.

| 파일 | 역할 |
|---|---|
| `app.py` | Streamlit 웹 UI, 사이드바 옵션, 탭별 결과 렌더링, 시각화 구성 |
| `src/member_activity_workflow.py` | LangGraph 워크플로우, API 호출, 캐시, LLM 분석, 표결·정당 일치도 계산 |
| `src/config.py` | API 키, 캐시 경로, 환경 변수 관리 |
| `src/member_directory.py` | 의원 목록 브라우저 및 정당 매칭용 의원 디렉토리 처리 |
| `src/streamlit_runtime.py` | Streamlit 캐시 리소스 기반 클라이언트·그래프 재사용 |

주요 기술스택은 Python, Streamlit, LangGraph, LangChain Core, LangChain Google GenAI, pandas, requests, python-dotenv이다. 시각화에는 Streamlit 기본 UI와 ECharts, vis-network CDN을 활용한다. LLM 모델은 `gemini-2.5-flash-lite`로 고정한다.

### 4.2 LangGraph 워크플로우 설계

본 프로젝트는 단순한 LLM 질의응답이 아니라, 데이터 수집과 분석 순서가 명확한 에이전트형 워크플로우이다. 따라서 LLM이 임의로 도구 호출 순서를 정하는 방식이 아니라, LangGraph의 `StateGraph(MemberActivityState)`로 노드와 의존 관계를 코드 수준에서 정의한다.

워크플로우는 다음과 같이 작동한다.

1. `normalize_user_request`: 사용자 입력과 옵션을 정규화한다.
2. `get_member_info`: 의원 기본정보와 재임 대수를 조회한다.
3. `get_member_info` 이후 병렬 fan-out:
   - `search_member_bills`: 발의법안 조회
   - `get_all_member_votes`: 표결 상세 조회
   - `search_recent_member_news`: 최근 뉴스 검색
4. `search_member_bills` 이후 병렬 분석:
   - `analyze_legislative_interests`: 입법 관심 분야 분석
   - `analyze_cosponsor_network`: 공동발의 네트워크 분석
5. `get_all_member_votes` 이후 순차 분석:
   - `analyze_party_alignment`: 정당 다수 입장 일치도 계산
   - `interpret_vote_statistics`: 표결 통계 해석 메모 생성
6. fan-in:
   - 입법 관심 분야, 공동발의 네트워크, 표결 해석, 최근 뉴스 결과가 모두 준비된 뒤 `generate_activity_briefing`에서 핵심 브리핑을 생성한다.
7. `summarize_member_activity`: 최종 요약 결과를 생성한다.

병렬화 기준은 서로의 산출물을 기다릴 필요가 없는 작업만 동시에 실행하는 것이다. 예를 들어 발의법안 조회와 표결 상세 조회는 모두 의원 기본정보만 필요하므로 동시에 실행 가능하다. 반면 정당 다수 입장 일치도는 표결 상세 조회 결과가 있어야 계산 가능하므로 표결 조회 이후에만 실행한다.

병렬 노드들이 동시에 오류 메시지나 LLM quota 상태를 갱신할 수 있으므로, 상태 병합에는 reducer를 적용한다. `errors`는 `merge_unique_messages`로 중복 없이 병합하고, `llm_quota_exhausted`는 `bool_or`로 병합한다.

### 4.3 주요 분석 기능

#### 4.3.1 의원 기본정보 및 재임 대수 필터링

사용자가 의원명을 입력하면 먼저 국회의원 정보 통합 API에서 의원 기본정보를 조회한다. 이때 정당, 선거구, 위원회, 재임 대수를 확인하고, 이후 분석 범위를 해당 의원이 실제 재임한 대수로 제한한다. 이를 통해 모든 대수를 무작정 조회하는 비효율을 줄이고, 특정 의원이 활동하지 않은 대수를 분석 대상으로 포함하는 오류를 방지한다.

#### 4.3.2 발의법안 분석과 입법 관심 분야 해석

발의법안 분석은 의원별 발의법안 API를 이용한다. 대표발의와 공동발의를 분류하고, 처리결과별 통계, 소관위원회별 통계를 계산한다. LLM은 발의법안 목록과 위원회 분포를 바탕으로 의원의 입법 관심 분야를 요약하되, 정치적 호불호나 성과 평가를 하지 않고 통계와 법안명에 근거한 조심스러운 해석만 생성하도록 설계한다.

#### 4.3.3 공동발의 네트워크 분석

공동발의 네트워크는 대상 의원과 같은 법안에 이름을 올린 의원들을 연결하는 1차 ego-network로 구성한다. 네트워크에서는 공동발의 빈도에 따라 노드 크기를 조정하고, 협업 관계를 당내·당외·정당 미확인으로 구분한다. 또한 반복 협업 의원 Top 10, 협업 소관위원회, 초당성 유형을 제시한다. 이 분석은 공동발의가 실제 정책 동의나 친소관계를 직접 증명한다는 뜻이 아니며, 법안 발의 과정에서 함께 이름을 올린 입법 협력 관계를 탐색하는 지표로 사용한다.

#### 4.3.4 표결 상세 조회

표결 상세 조회는 `ncocpgfiaoituanbr`와 `nojepdqqaweusdfbi`를 결합한다. 의안별 표결현황 API에서 표결 대상 의안 목록과 `BILL_ID`를 확보하고, 국회의원 본회의 표결정보 API에서 해당 의원의 찬성·반대·기권·불참 기록을 수집한다. 베타 버전의 기본 표결 상세 범위는 최근 100건이며, 사용자가 세부 설정에서 조회 범위를 조정할 수 있다.

표결 조회는 먼저 의원명 기준 일괄 조회를 시도하고, 부족한 경우 `BILL_ID`별 조회로 보완한다. 이 과정은 시간이 오래 걸릴 수 있으므로 캐시를 적극 사용한다. API 호출 실패가 일부 발생해도 전체 워크플로우가 중단되지 않도록 성공·실패 건수를 분리하고, 실패분은 다음 실행에서 재시도할 수 있도록 설계한다.

#### 4.3.5 정당 다수 입장 일치도 분석

정당 다수 입장 일치도는 공식 당론이 아니다. 본 프로젝트는 각 표결에서 같은 정당 의원들이 실제로 가장 많이 선택한 표결값을 정당 다수 입장으로 계산한다. 찬성·반대·기권 중 최다값을 다수 입장으로 보고, 불참은 다수 입장 계산에서 제외한다. 대상 의원의 표결이 정당 다수 입장과 같으면 일치, 다르면 이탈, 대상 의원이 불참한 경우 불참으로 분류한다. 무소속, 정당 내 유효 표결 없음, 찬성·반대·기권 동률 등은 판정 제외로 분리한다.

이 분석은 의원 개인의 표결 행동을 정당정치의 맥락 속에서 이해하기 위한 보조 지표이다. 사용자는 특정 의원이 같은 정당의 다수 흐름과 얼마나 유사하게 표결했는지, 어느 표결에서 이탈했는지, 불참 비중은 어느 정도인지 확인할 수 있다.

#### 4.3.6 최근 이슈 요약과 핵심 브리핑

최근 이슈는 Google News RSS에서 검색한 기사 메타데이터를 바탕으로 요약한다. LLM은 제공된 뉴스 검색 결과만 근거로 요약하며, 기사 내용의 사실관계나 의원의 의도를 단정하지 않도록 제한한다. 핵심 브리핑은 의원 프로필, 발의법안 통계, 공동발의 네트워크, 표결 요약, 정당 일치도 요약, 최근 뉴스 수를 바탕으로 첫 화면 요약을 제공한다.

### 4.4 LLM 프롬프트 설계 원칙

본 프로젝트의 LLM 분석은 해석을 보조하는 역할에 한정된다. 모든 system prompt는 다음 원칙을 따른다.

- 제공된 JSON 통계와 API 조회 결과만 근거로 분석한다.
- 정치적 호불호, 성과 평가, 의도 추정은 하지 않는다.
- 공식 의정활동 데이터와 웹 검색 기반 데이터를 구분한다.
- 추정이 필요한 경우 단정하지 않고 조심스럽게 표현한다.
- LLM 호출 실패 또는 quota 초과 시 규칙 기반 fallback을 사용한다.

이러한 설계는 LLM이 데이터에 없는 정치적 의미를 과잉 생성하는 것을 막기 위한 장치이다. 특히 의정활동 데이터는 정치적으로 민감할 수 있으므로, 분석 결과는 평가나 단정이 아니라 사용자의 탐색과 판단을 돕는 근거 중심 요약으로 제한한다.

### 4.5 캐시와 오류 격리 전략

베타 버전은 API 호출 비용과 실행 시간을 줄이기 위해 캐시를 사용한다. 의원 목록은 Streamlit 캐시를 사용하고, 발의법안, 표결 의안 목록, 의원별 표결, BILL_ID별 정당 표결 분포, LLM 분석 결과, 최근 뉴스 검색 결과는 파일 캐시로 저장한다. 기본 캐시 TTL은 24시간이며, 최근 뉴스 캐시는 6시간 기준으로 관리한다.

오류 격리도 중요한 설계 요소이다. 표결 500건 중 일부 API 호출이 실패해도 전체 워크플로우가 죽지 않고 성공·실패 건수를 분리해 표시한다. 사용자는 부분 결과를 확인할 수 있고, 다음 실행에서는 실패한 표결만 재시도할 수 있다.

## 5. 베타 버전 구현 현황

현재 베타 버전은 Streamlit 앱으로 구현되어 있으며, 사용자가 직접 Gemini API 키를 입력해 LLM 분석을 사용할 수 있다. 열린국회정보 API 키는 서버 측 설정 또는 `.env`를 통해 관리하며, 사용자에게 노출하지 않는 구조를 권장한다.

베타 버전에서 구현된 주요 화면은 다음과 같다.

- **요약 탭**: 의원 프로필, KPI, 핵심 브리핑, 발의법안 요약, 입법 관심 분야 분석
- **발의 법안 탭**: 발의법안 통계, 공동발의 네트워크, 반복 협업 의원, 법안 원자료
- **표결 탭**: 찬성·반대·기권·불참 분류, 표결 요약, 표결 상세 원자료 CSV 다운로드
- **정당 일치도 탭**: 일치·이탈·불참·판정 제외 요약, 이탈 표결 목록, 분석 방식 안내, 범위 재분석
- **최근 이슈 탭**: Google News RSS 기반 최근 기사 목록, 뉴스 타임라인, LLM 요약
- **원자료 탭**: 워크플로우 결과 JSON, 오류 및 확인 메시지

또한 사이드바에는 분석 옵션, 세부 조회 범위, 의원 목록 브라우저, 이론 배경 설명을 제공한다. 이론 배경에는 의정활동의 개념, 의정활동 기록의 의미, 공론장과 정치 커뮤니케이션, 감시저널리즘과 민주적 책임성, 국회의원의 복합적 지위, 정당 다수 입장 일치도의 중요성이 포함된다.

## 6. 평가 기준

### 6.1 참신성

본 프로젝트의 참신성은 단일 API 조회가 아니라 **의원 중심 통합 분석**을 제공한다는 점에 있다. 국회의원 기본정보, 발의법안, 본회의 표결, 정당 다수 입장 일치도, 공동발의 네트워크, 최근 이슈를 하나의 워크플로우 안에서 연결한다. 특히 정당 다수 입장 일치도는 같은 정당 의원들의 실제 표결 분포를 계산해 의원의 표결 행동을 정당정치의 맥락에서 볼 수 있도록 한다는 점에서 차별성이 있다.

또한 LangGraph를 활용해 분석 단계를 노드로 분리하고, 서로 독립적인 작업은 병렬 처리하는 구조를 구현했다. 이는 단순한 챗봇형 응답이 아니라, 공공데이터 수집·계산·해석·시각화를 결합한 데이터 에이전트라는 점에서 참신하다.

### 6.2 필요성

의정활동 데이터는 공개되어 있지만 흩어져 있고, 시민이 직접 해석하기 어렵다. 민주주의에서 시민의 정치적 판단과 언론의 감시 기능이 작동하려면 공적 정보가 접근 가능하고 이해 가능한 형태로 제공되어야 한다. 본 프로젝트는 국회의원 의정활동 데이터를 통합하고, 통계·차트·네트워크·LLM 요약을 통해 사용자가 빠르게 맥락을 파악할 수 있도록 한다. 시민에게는 정치 판단의 근거를 제공하고, 언론인에게는 감시저널리즘의 취재 단서를 제공한다는 점에서 필요성이 높다.

### 6.3 가능성

베타 버전이 이미 구현되어 있어 실현 가능성은 높다. 열린국회정보 API와 Google News RSS를 실제로 호출하고, Streamlit 웹 앱으로 배포 가능한 구조를 갖추었다. 캐시와 병렬 조회, 오류 격리, LLM fallback 전략도 구현되어 있어 API 지연이나 LLM 호출 실패 상황에서도 일정 수준의 안정성을 확보했다. 향후 날짜 기반 표결 조회, 의원 비교 기능, 정책 키워드 기반 검색 기능 등을 추가하면 확장 가능성이 크다.

## 7. 베타 버전 이후 발전 및 개선 방향

### 7.1 표결 상세 조회의 날짜 기준 전환

현재 베타 버전의 표결 상세 조회는 최근 N건 기준으로 동작하며, 기본값은 최근 100건이다. 사용자가 세부 설정에서 조회 범위를 조정할 수 있지만, 최근 N건 방식에는 구조적 한계가 있다.

첫째, 본회의 표결은 하루에 매우 많이 이루어질 수 있다. 예를 들어 어느 날 90건의 표결이 있고, 다음 날 80건의 표결이 있었다고 하자. 사용자가 최근 100건을 조회하면 첫째 날 90건과 둘째 날 10건만 포함된다. 이 경우 둘째 날의 표결 맥락은 10건만 반영되어, 하루 단위 본회의 흐름을 온전히 파악하기 어렵다.

둘째, 의원이 특정 날짜의 본회의 표결에 전부 불참한 경우, 의원명 기준 표결 조회에서는 해당 날짜가 충분히 드러나지 않을 수 있다. 즉, “의원이 표결한 기록”만 기준으로 보면 불참이 많았던 본회의 날짜를 놓칠 수 있다. 불참 역시 중요한 의정활동 정보이므로, 표결 분석은 단순히 최근 N개 표결을 자르는 방식보다 날짜 단위로 접근하는 것이 더 적절할 수 있다.

따라서 향후 개선 방향은 **날짜 기반 표결 조회**이다. 구체적인 개선안은 다음과 같다.

1. `의안별 표결현황` API에서 표결이 있었던 본회의 날짜 목록을 생성한다.
2. 사용자가 특정 의원을 입력하면, 해당 의원의 표결 API 지원 재임 대수를 확인한다.
3. 사용자는 “최근 N건 기준”과 “본회의 날짜 기준” 중 조회 방식을 선택할 수 있다.
4. 날짜 기준을 선택하면, 본회의 표결일 목록에서 하루 또는 여러 날짜를 선택한다.
5. 선택한 날짜의 전체 `BILL_ID`를 조회하고, 각 의안에 대해 해당 의원의 찬성·반대·기권·불참을 계산한다.
6. 해당 의원의 표결 행이 없고 해당 날짜에 의안 표결이 있었다면 불참으로 분류한다.
7. 정당 다수 입장 일치도도 선택 날짜의 의안 전체를 기준으로 다시 계산한다.

이 방식은 최근 N건 방식보다 해석력이 높다. 사용자는 “2026년 5월 7일 본회의에서 이 의원은 몇 건에 찬성·반대·기권·불참했는가”처럼 하루 단위의 정치적 맥락을 볼 수 있다. 또한 특정 본회의 날짜에 표결이 몰려 있을 때도 분석 범위가 임의로 잘리지 않는다.

### 7.2 날짜 후보 UI 설계

날짜 기반 조회를 구현할 때는 사용자 경험을 고려해야 한다. 분석 시작 전 사이드바 세부 설정에서 다음과 같은 흐름을 제공할 수 있다.

```text
표결 조회 방식
- 최근 N건 기준
- 본회의 날짜 기준

날짜 후보 불러오기
- 의원 참여일만 보기
- 전체 본회의 표결일 보기

조회할 날짜 선택
- 2026-05-07: 표결 의안 90건
- 2026-04-30: 표결 의안 50건
- 2026-04-29: 표결 의안 12건
```

기본값은 현재처럼 최근 N건 기준으로 유지하는 것이 좋다. 날짜 후보를 자동으로 불러오면 초기 앱 속도가 느려질 수 있으므로, 사용자가 “날짜 후보 불러오기” 버튼을 눌렀을 때만 API를 호출하는 방식이 적절하다.

### 7.3 추가 고도화 방향

날짜 기반 표결 조회 외에도 다음 기능을 발전 방향으로 고려할 수 있다.

- **의원 간 비교 기능**: 같은 대수나 같은 정당 의원들끼리 발의법안 수, 표결 참여율, 정당 일치도, 공동발의 네트워크를 비교한다.
- **정책 키워드 기반 분석**: AI, 청년, 부동산, 노동 등 특정 키워드와 관련된 발의법안과 표결을 의원별로 추적한다.
- **위원회 활동 연계**: 소관위원회와 발의법안·표결 데이터를 연결해 의원의 상임위 활동과 입법 관심 분야를 더 정교하게 분석한다.
- **뉴스 관련성 필터 고도화**: 최근 뉴스 검색 결과 중 의정활동과 무관한 기사나 단순 사진 기사를 제외하는 relevance scoring을 도입한다.
- **재현 가능한 분석 리포트 생성**: 특정 의원 분석 결과를 Markdown, PDF, CSV 묶음으로 내보내 시민단체나 언론인이 기록으로 보관할 수 있게 한다.
- **API 제공 범위 안내 강화**: 20대 이전 표결정보처럼 API 제공 범위 밖인 경우를 단순 “조회 결과 없음”과 명확히 구분해 사용자에게 안내한다.

## 8. 기대 효과

본 프로젝트는 국회의원 의정활동을 데이터 기반으로 감시하고 이해할 수 있는 기반을 제공한다. 시민은 한 의원의 발의법안과 표결을 빠르게 확인하고, 정당 다수 입장과의 관계를 통해 표결 행동의 정치적 맥락을 살펴볼 수 있다. 언론인은 반복적인 공동발의 관계, 특정 분야에 집중된 발의 활동, 높은 불참 비중, 정당 다수 입장 이탈 표결을 취재 단서로 활용할 수 있다.

기술적으로는 LangGraph를 활용해 공공데이터 수집, 병렬 처리, 캐시, 오류 격리, LLM 기반 해석을 결합한 실용적 AI 에이전트 사례를 제시한다. 사회적으로는 공론장의 질을 높이는 정보 매개 장치이자, 감시저널리즘과 민주적 책임성을 지원하는 데이터 기반 정치 커뮤니케이션 서비스로 기능할 수 있다.
