# LangGraph 기반 워크플로우 선택 이유


 "LLM 프롬프트만으로 에이전트 흐름을 제어하면 불안정하고, 도구 실행 결과를 관찰하기 어렵다"는 점이다. 우리 프로젝트의 국회의원 의정활동 추적 워크플로우는 이 문제를 해결하기 위해 LangGraph를 사용한다. 

기존 ReAct 또는 AgentExecutor 방식이라면 LLM에게 다음과 같은 역할을 프롬프트로 지시해야 한다.

```text
의원 이름을 받으면 의원 기본정보를 조회하고,
재임 대수를 확인한 뒤,
발의법안을 조회하고,
표결정보를 조회하고,
정당 다수 입장 일치도를 계산하고,
최근 뉴스까지 검색해서 요약하라.
``` 

이 방식은 LLM이 순서를 지키지 않거나, 중간 도구 호출을 생략하거나, 실패한 API 결과를 제대로 반영하지 못할 위험이 있다. 반면 우리 프로젝트는 LangGraph의 `StateGraph`를 사용해 각 단계를 명시적인 노드로 분리했다.

현재 워크플로우는 다음과 같은 순서로 코드 수준에서 강제된다.

```text
normalize_user_request
-> get_member_info
-> search_member_bills
-> analyze_legislative_interests
-> analyze_cosponsor_network
-> get_all_member_votes
-> analyze_party_alignment
-> interpret_vote_statistics
-> search_recent_member_news
-> generate_activity_briefing
-> summarize_member_activity
```

따라서 LLM이 임의로 "발의법안 조회를 건너뛰고 답변한다"거나 "표결정보를 조회하기 전에 정당 일치도를 계산한다"는 흐름이 발생하지 않는다. 분석 순서가 프롬프트가 아니라 Python 코드의 그래프 구조로 고정되어 있기 때문이다.

## 프롬프트 제어 한계 극복

우리 서비스는 단순 질의응답이 아니라 여러 API와 분석 단계를 조합한다. 특히 국회의원 이름 하나를 입력받아도 다음 판단이 필요하다.

- 해당 의원이 실제로 재임한 대수를 확인한다.
- 발의법안 API가 지원하는 대수와 지원하지 않는 대수를 구분한다.
- 표결 API가 지원하는 재임 대수 중 가장 최근 대수만 초기 조회한다.
- 표결 API 지원 범위 밖의 대수는 "조회 결과 없음"이 아니라 "API 제공 범위 밖"으로 안내한다.
- 정당 일치도 분석은 공식 당론이 아니라 같은 정당 의원 다수 표결값을 기준으로 계산한다.

이 로직을 전부 프롬프트에 맡기면 모델이 조건을 누락하거나 설명을 섞을 가능성이 크다. LangGraph를 사용하면 각 조건을 노드 내부 코드로 처리할 수 있고, 다음 노드로 넘길 값은 `MemberActivityState`에 명확히 저장된다.

예를 들어 `get_member_info` 노드는 의원 기본정보와 재임 대수를 확인하고, 이후 노드가 사용할 `analysis_terms`, `bill_terms`, `vote_terms`, `unsupported_vote_terms`를 상태에 기록한다. 이후 발의법안 노드와 표결 노드는 이 상태값을 기준으로만 실행된다.

## 도구 실행 관찰과 디버깅

"툴 실행 확인의 어려움"도 우리 프로젝트에서 중요한 문제였다. 열린국회정보 API는 엔드포인트별 제공 범위가 다르고, 일부 대수는 데이터가 없어서 빈 결과가 나오는 것이 아니라 API 제공 범위 자체가 제한된다.

LangGraph를 사용하면서 각 도구 실행 결과를 독립된 노드 단위로 확인할 수 있게 되었다.

- `get_member_info`: 의원 기본정보와 재임 대수 확인
- `search_member_bills`: 대표발의, 공동발의, 처리결과별 통계 생성
- `get_all_member_votes`: 찬성, 반대, 기권, 불참 표결 분류
- `analyze_party_alignment`: 정당 다수 입장과의 일치/이탈/불참 계산
- `search_recent_member_news`: 최근 뉴스 검색 및 요약

또한 각 노드는 `progress_log`를 통해 진행 상황을 남기고, Streamlit 화면에서도 현재 단계와 세부 진행을 표시한다. 사용자는 단순히 "분석 중"이라는 메시지만 보는 것이 아니라, 어느 단계에서 시간이 걸리는지 확인할 수 있다.

이 구조는 디버깅에도 유리하다. 예를 들어 표결이 모두 불참으로 분류되는 문제가 있었을 때, 표결 조회 노드와 정당 일치도 분석 노드를 분리해서 원인을 추적할 수 있었다. API 호출 결과, 캐시 사용 여부, BILL_ID별 상세 조회 성공/실패 건수를 각각 확인할 수 있었기 때문이다.

## 상태 기반 데이터 재사용

LangGraph를 선택한 또 다른 이유는 중간 결과를 상태로 축적하고 재사용하기 쉽기 때문이다.

우리 프로젝트는 한 번의 분석에서 다음 데이터를 상태에 저장한다.

- 의원 기본정보
- 발의법안 목록
- 발의법안 처리결과별 통계
- 발의법안 소관위원회별 통계
- 의원별 표결 기록
- BILL_ID별 정당 표결 분포
- 정당 다수 입장 일치도 결과
- 최근 뉴스 검색 결과

이 데이터는 이후 노드에서 다시 사용된다. 예를 들어 정당 일치도 분석은 앞서 확보한 의원 표결 기록을 fallback 데이터로 활용하고, 표결 해석 메모는 정당 일치도 요약과 날짜 범위를 기반으로 생성된다.

또한 반복 조회 속도를 줄이기 위해 파일 기반 캐시를 함께 사용했다. 표결 의안 목록, 의원별 표결 기록, BILL_ID별 정당 표결, 발의법안 조회 결과, 뉴스 검색 결과, LLM 분석 결과를 캐시해 동일 조건의 재조회에서 API 호출을 줄인다.

## 오류 격리와 안정성

현업형 워크플로우에서는 일부 API나 LLM 호출이 실패해도 전체 분석이 중단되지 않아야 한다. 우리 프로젝트는 이 점에서도 LangGraph 구조의 장점을 활용했다.

예를 들어 Gemini 호출이 quota 초과로 실패하면 전체 워크플로우를 종료하지 않고 규칙 기반 요약으로 대체한다. 표결 상세 조회 중 일부 BILL_ID 호출이 실패해도 전체 결과에는 "성공 n건, 실패 n건"처럼 실패 건수를 분리해 표시한다.

즉, 하나의 단계 실패가 전체 앱 실패로 이어지지 않도록 노드별 오류 처리를 적용했다.  "툴 실행 관찰과 예외 처리의 어려움"을 코드 구조로 해결한 부분이다.

## 왜 LangGraph를 선택했는가

이 프로젝트에서 LangGraph를 선택한 이유는 단순히 최신 프레임워크를 사용하기 위해서가 아니다. 국회의원 의정활동 추적은 순서가 중요한 데이터 파이프라인이다.

의원 기본정보를 알아야 재임 대수를 알 수 있고, 재임 대수를 알아야 조회할 발의법안과 표결 대수를 제한할 수 있다. 표결 정보를 확보해야 정당 다수 입장 일치도를 계산할 수 있고, 그 결과를 바탕으로 표결 해석 메모를 만들 수 있다.

따라서 이 프로젝트는 LLM이 자유롭게 도구를 선택하는 자율 에이전트보다, 개발자가 분석 순서와 상태 전이를 명확히 통제하는 그래프 워크플로우가 더 적합하다.

LangGraph를 사용함으로써 다음 장점을 얻었다.

- 분석 순서를 코드로 강제해 프롬프트 의존성을 줄였다.
- API 실행 결과를 노드별로 관찰하고 디버깅할 수 있게 했다.
- 중간 결과를 `State`에 저장해 다음 분석 단계에서 재사용했다.
- API 제공 범위 밖 데이터와 실제 조회 결과 없음 상태를 구분했다.
- 캐시, 병렬 처리, fallback, 오류 격리를 워크플로우 안에 안정적으로 넣었다.
- Streamlit 화면에서 진행 상황을 단계별로 보여줄 수 있게 했다.

## 요약 문장

이 프로젝트는 국회의원 이름 하나를 입력하면 의원 기본정보, 발의법안, 표결 기록, 정당 다수 입장 일치도, 최근 뉴스 이슈를 순차적으로 분석하는 LangGraph 기반 워크플로우이다.

LangGraph를 선택한 이유는 기존 프롬프트 기반 에이전트처럼 LLM의 판단에 흐름을 맡기지 않고, 의원정보 조회부터 표결 분석까지의 순서를 코드 수준에서 강제하기 위해서다. 또한 각 API 호출과 분석 결과를 노드별로 관찰할 수 있어 디버깅, 캐싱, 오류 격리, 진행률 표시를 안정적으로 구현할 수 있었다.

즉, 우리 프로젝트에서 LangGraph는 단순한 에이전트 프레임워크가 아니라 복잡한 의정활동 분석 절차를 신뢰성 있게 실행하기 위한 워크플로우 제어 장치로 사용되었다.

## 추가, 사용한 열린국회정보 API

우리 프로젝트는 열린국회정보의 여러 Open API 목록 중 아래 4개 서비스를 사용한다.

| 열린국회정보 Open API 서비스명 | API 코드 | 프로젝트에서 사용하는 정보 |
|---|---:|---|
| 국회의원 정보 통합 API | `ALLNAMEMBER` | 의원 기본정보, 정당, 선거구, 위원회, 재임 대수 |
| 국회의원 발의법률안 | `nzmimeepazxkubdpn` | 발의법안 목록, 대표발의/공동발의, 처리결과, 소관위원회 |
| 의안별 표결현황 | `ncocpgfiaoituanbr` | 본회의 표결 의안 목록, `BILL_ID`, 의안명, 표결일 |
| 국회의원 본회의 표결정보 | `nojepdqqaweusdfbi` | 의원별 찬성/반대/기권/불참 표결, 정당별 표결 분포 계산용 원자료 |

`국회기록원 연차보고서`, `국회기록원 정보 목록` 같은 기록원 계열 API는 이 프로젝트에서 사용하지 않는다.

데이터 흐름은 다음과 같다.

```text
국회의원 정보 통합 API
-> 의원 기본정보와 재임 대수 확인

국회의원 발의법률안
-> 발의법안 목록 조회
-> 처리결과별 통계와 소관위원회별 통계 계산

의안별 표결현황
-> 본회의 표결 의안 목록과 BILL_ID 확보

국회의원 본회의 표결정보
-> 의원별 표결 기록 조회
-> 같은 정당 의원들의 표결 분포 계산
-> 정당 다수 입장 일치/이탈 분석 생성
```

최근 의원 관련 이슈는 열린국회정보 API가 아니라 Google News RSS에서 가져온다. Gemini는 데이터를 가져오는 API가 아니라, 이미 수집한 발의법안·표결·뉴스 데이터를 요약하고 해석하는 데 사용한다.

결과 항목별 출처는 다음과 같이 정리할 수 있다.

| 결과 항목 | 출처 |
|---|---|
| 의원 기본정보 | `ALLNAMEMBER` 원자료 |
| 발의법안 목록 | `nzmimeepazxkubdpn` 원자료 |
| 발의법안 처리결과별 통계 | 발의법안 목록을 기준으로 우리 코드가 계산 |
| 발의법안 소관위원회별 통계 | 발의법안 목록을 기준으로 우리 코드가 계산 |
| 의원별 표결 기록 | `nojepdqqaweusdfbi` 원자료 |
| `BILL_ID`별 정당 표결 분포 | `ncocpgfiaoituanbr`로 `BILL_ID`를 확보한 뒤, `nojepdqqaweusdfbi`로 정당별 표결을 조회해 구성 |
| 정당 다수 입장 일치도 결과 | `BILL_ID`별 정당 표결 분포를 기준으로 우리 코드가 계산 |
| 최근 뉴스 검색 결과 | Google News RSS |
